ПРОЕКТ ЕЩЕ В РАБОТЕ

CNN для классификации статей Habr.com

Источник данных
Сайт: habr.com; раздел AI и ML -> Машинное обучение: habr.com/ru/hubs/machine_learning/articles/

Способ сбора: библиотека Playwright (headless Chromium) + парсинг HTML-страниц статей и пагинации (BeautifulSoup).

Разметка 
Автоматическая: через React-артефакт Сlaude — берёт статьи из JSON-файла, отправляет их в Claude API батчами и сохраняет результат.

Разметка по уровням сложности: простой(0), средний(1), продвинутый(2). Определяется по тегу на Habr.

Токенизация - word-level + character-level (тексты строгие, объемные и выверенные, так как это статьи, но содержат много профессионального сленга)

Архитектура модели:
TextCNN с одномерными свертками с ядрами разного размера (нужно, чтобы улавливать n-граммы разной длины).

### Гиперпараметры конфигурации 

| Параметр | Тип / Значение | Описание | Обоснование для текстов Хабра |
| :--- | :--- | :--- | :--- |
| **`vocab_size`** | `int` (зависит от датасета) | Размер словаря токенизатора | Рекомендуется ограничить top-50000 токенами, заменив редкие слова на `[UNK]`. |
| **`embedding_dim`** | `300` | Размерность вектора эмбеддинга | Стандарт для качественных предобученных векторов (например, *FastText* или *Word2Vec* от RusVectores). |
| **`filter_sizes`** | `[3, 4, 5]` | Размеры ядер свёртки (`kernel_size`) | Извлекают n-граммы разной длины: триграммы, 4-граммы и 5-граммы (ключевые фразы и термины). |
| **`n_filters`** | `128` | Количество фильтров на каждый размер ядра | Количество извлекаемых паттернов. Значение 128 обеспечивает баланс между точностью и скоростью. |
| **`dropout`** | `0.5` | Коэффициент регуляризации Dropout | Высокое значение критично для предотвращения переобучения на длинных и зашумленных текстах. |
| **`output_dim`** | `3` | Количество выходных классов | Соответствует уровням сложности статей: `0` — Простой, `1` — Средний, `2` — Сложный. |
| **`pad_idx`** | `int` | Индекс токена заполнения (`[PAD]`) | Используется для выравнивания длины батчей, градиенты через этот токен не считаются. |

Почему именно эта архитектура хороша для Хабра?
Устойчивость к длине текста: Статьи на Хабре бывают как короткими заметками, так и лонгридами. Слой max_pool1d схлопывает временную размерность (seq_len), превращая текст любой длины в фиксированный вектор.
Скорость: В отличие от RNN (LSTM/GRU), CNN обрабатывает слова параллельно, что делает обучение в разы быстрее. Это критично, если вы обучаете модель на CPU или слабых GPU.
Выделение триггеров сложности: Модель отлично выцепляет маркеры сложности. Например, появление фраз "математическое ожидание", "ассемблерный код", "memory leak" во многом определит "хардкорность" статьи быстрее, чем это сделает рекуррентная сеть, анализирующая весь контекст последовательно.
